## 一、技术选型确认

### 1.1 语言和框架

| 类别 | 选型 | 版本 |
|------|------|------|
| 语言 | Python | 3.11+ |
| Web 框架 | FastAPI | 0.110+ |
| ASGI 服务器 | Uvicorn | 0.27+ |
| 数据验证 | Pydantic v2 + pydantic-settings | 2.5+ |
| ORM | SQLAlchemy 2.0（异步） | 2.0+ |
| 数据库 | **开发环境：SQLite（WAL 模式）/ 生产环境：PostgreSQL 15+** | - |
| 数据库迁移 | Alembic | 1.13+ |
| 任务队列 | Celery + Redis（第二版） | - |
| LLM 客户端 | OpenAI SDK（DeepSeek 兼容） | 1.10+ |

> **技术栈变更说明（2026-07-10 修宪同步）**：MVP 阶段将数据库从 PostgreSQL 改为「开发用 SQLite / 生产用 PostgreSQL」双环境策略，降低本地开发门槛。ORM 统一为 SQLAlchemy 2.0（async），兼容 SQLite 和 PostgreSQL。后端业务层从「Next.js API Routes + Prisma」调整为「FastAPI 统一负责业务编排 + 计算层」，Next.js API Routes 保留为 BFF 层。

### 1.2 选择原因

| 选型 | 为什么选 |
|------|---------|
| **Python 3.11+** | 统计库（scipy/statsmodels/factor_analyzer/pingouin）只有 Python 生态；3.11 性能提升 10-60%；类型提示更完善 |
| **FastAPI** | 异步原生（async/await）；自动 OpenAPI 文档；Pydantic 深度集成；类型安全；性能接近 Node.js/Go |
| **Pydantic v2** | FastAPI 原生集成；数据验证 + 序列化；Rust 核心（pydantic-core）性能高 |
| **SQLAlchemy 2.0** | Python ORM 事实标准；2.0 版本支持异步；类型提示完善；兼容 SQLite 和 PostgreSQL |
| **Alembic** | SQLAlchemy 官方迁移工具；版本控制数据库变更 |
| **SQLite（开发环境）** | 零配置；单文件部署；降低开发门槛；WAL 模式支持并发读写 |
| **PostgreSQL（生产环境）** | 并发支持好；JSON 字段支持；扩展性强；与 SQLAlchemy 无缝切换 |

### 1.3 不选什么、为什么

| 不选 | 理由 |
|------|------|
| **Flask** | 同步框架，异步支持弱；无自动文档；数据验证需第三方库 |
| **Django** | 过重；ORM 不如 SQLAlchemy 灵活；异步支持晚 |
| **Node.js/Express** | 统计库生态差；scipy/statsmodels 无 Node 版本 |
| **Go/Gin** | 统计库缺失；开发效率低；不适合数据科学场景 |
| **MySQL** | JSON 支持弱；窗口函数支持晚；扩展性不如 PostgreSQL |
| **MongoDB** | 事务支持弱；关系型数据不适合；与 SQLAlchemy 不兼容 |

## 二、Python 工程规范

### 2.1 依赖管理

| 规范 | 说明 |
|------|------|
| **依赖锁定** | 使用 `requirements.txt` 锁定版本（`pip freeze`）；第二版可迁移至 `poetry` |
| **虚拟环境** | 必须使用虚拟环境（`venv` 或 `conda`）；禁止全局安装 |
| **依赖分类** | 区分运行时依赖（`requirements.txt`）和开发依赖（`requirements-dev.txt`） |
| **新增依赖** | 必须说明理由，更新本文档，并经 review 确认 |

### 2.2 配置读取

| 规范 | 说明 |
|------|------|
| **配置来源** | 环境变量 + `.env` 文件（开发环境）；生产环境用环境变量或密钥管理服务 |
| **配置类** | 使用 `pydantic-settings` 的 `BaseSettings`；统一在 `app/core/config.py` 定义 |
| **配置访问** | 通过 `settings.XXX` 访问；禁止硬编码密钥或魔法数字 |
| **敏感信息** | API Key、数据库密码等必须从环境变量读取；禁止提交到代码库 |

### 2.3 日志规范

| 规范 | 说明 |
|------|------|
| **日志库** | 使用标准库 `logging`；生产环境可切换 `structlog`（结构化日志） |
| **日志级别** | DEBUG（开发）/ INFO（生产）/ WARNING（异常）/ ERROR（错误）/ CRITICAL（严重） |
| **日志格式** | 开发：`%(asctime)s | %(levelname)-8s | %(name)s:%(lineno)d | %(message)s`；生产：JSON 格式 |
| **日志位置** | 统一在 `app/core/logging.py` 配置；各模块通过 `logging.getLogger(__name__)` 获取 |
| **禁止事项** | 禁止 `print()`；禁止日志中输出敏感信息（密码、Token） |

### 2.4 异常处理

| 规范 | 说明 |
|------|------|
| **异常类** | 自定义业务异常继承 `AppException`（在 `app/core/exceptions.py` 定义） |
| **HTTP 异常** | 使用 FastAPI 的 `HTTPException`；或自定义异常 + 全局异常处理器 |
| **错误响应** | 统一格式（见第四节）；禁止返回原始堆栈信息 |
| **异常捕获** | 只在边界层（路由、中间件）捕获；业务层抛出异常，不吞掉 |
| **错误码** | 使用 HTTP 标准状态码；业务错误码在文档中定义，不随意扩展 |

### 2.5 测试规范

| 规范 | 说明 |
|------|------|
| **测试框架** | `pytest` + `pytest-asyncio`（异步测试） |
| **测试类型** | 单元测试（services）+ 集成测试（API）+ E2E 测试（关键流程） |
| **测试数据** | 使用 `factory_boy` 或 fixtures；禁止硬编码测试数据 |
| **覆盖率** | 目标 80%+；关键业务逻辑（数据生成、统计分析）必须 100% |
| **Mock** | 外部依赖（LLM、数据库）使用 `pytest-mock` 或 `unittest.mock` |

### 2.6 代码风格

| 规范 | 说明 |
|------|------|
| **格式化工具** | `ruff`（推荐）或 `black` + `isort` |
| **类型提示** | 强制 type hints；所有函数签名必须有类型注解 |
| **命名规范** | 变量/函数：snake_case；类：PascalCase；常量：UPPER_SNAKE_CASE |
| **文档字符串** | 公共函数/类必须有 docstring（Google 风格） |
| **行长度** | 最大 100 字符（ruff/black 配置） |

## 三、FastAPI 最佳实践

### 3.1 路由组织

| 实践 | 说明 |
|------|------|
| **路由分组** | 使用 `APIRouter` 按业务域分模块（`api/v1/questionnaire.py`） |
| **路由前缀** | 版本前缀 `/api/v1`；业务域前缀 `/questionnaire`、`/simulation`、`/report` |
| **标签分组** | 使用 `tags` 参数分组（`@router.get("/", tags=["问卷体检"]）`） |
| **路由聚合** | 在 `api/v1/__init__.py` 中 include 所有子路由 |

### 3.2 依赖注入

| 实践 | 说明 |
|------|------|
| **依赖管理** | 使用 `Depends()` 管理数据库连接、认证、权限 |
| **依赖复用** | 公共依赖（如 `get_current_user`）提取到 `app/core/dependencies.py` |
| **异步依赖** | 数据库连接、HTTP 客户端等 I/O 操作必须用 `async def` |
| **依赖覆盖** | 测试时使用 `app.dependency_overrides` 替换依赖 |

### 3.3 数据验证

| 实践 | 说明 |
|------|------|
| **Schema 定义** | 使用 Pydantic `BaseModel`；在 `schemas/` 目录按业务域分文件 |
| **请求体验证** | 使用 `Body()` 或 Schema 类；禁止 `request.json()` 手动解析 |
| **响应模型** | 使用 `response_model` 参数；自动序列化 + 过滤字段 |
| **字段验证** | 使用 Pydantic 验证器（`Field`、`validator`、`root_validator`） |

### 3.4 异步优先

| 实践 | 说明 |
|------|------|
| **异步函数** | I/O 密集型操作（数据库、HTTP 调用）必须用 `async def` |
| **同步阻塞** | CPU 密集型操作（统计分析）使用 `run_in_threadpool` 或 Celery |
| **事件循环** | 禁止在异步函数中调用同步阻塞代码 |

### 3.5 文档和元数据

| 实践 | 说明 |
|------|------|
| **OpenAPI** | FastAPI 自动生成 `/docs`（Swagger UI）和 `/redoc` |
| **描述信息** | 为每个路由、操作、模型添加 `summary`、`description` |
| **示例数据** | 使用 `example` 或 `examples` 参数提供请求/响应示例 |

## 四、统一响应格式

### 4.1 成功响应

```json
{
  "code": 0,
  "message": "success",
  "data": { ... }
}
```

### 4.2 错误响应

```json
{
  "code": 40001,
  "message": "题目格式错误",
  "details": { ... }
}
```

### 4.3 错误码规范

| 范围 | 说明 |
|------|------|
| `0` | 成功 |
| `40000-49999` | 客户端错误（参数错误、权限不足） |
| `50000-59999` | 服务端错误（内部异常、第三方服务失败） |
| `60000-69999` | 业务错误（信效度不达标、生成失败） |

### 4.4 响应模型

```python
from pydantic import BaseModel
from typing import Any, Optional

class ResponseModel(BaseModel):
    code: int = 0
    message: str = "success"
    data: Optional[Any] = None

class ErrorResponse(BaseModel):
    code: int
    message: str
    details: Optional[dict] = None
```

## 六、数据库连接

### 6.1 连接方式

| 规范 | 说明 |
|------|------|
| **ORM** | SQLAlchemy 2.0（异步模式） |
| **连接池** | 使用 `create_async_engine` + `pool_size` + `max_overflow` |

| **连接字符串** | `postgresql+asyncpg://user:password@localhost:5432/dbname` |

### 6.2 会话依赖

```python
# app/core/database.py
from sqlalchemy.ext.asyncio import create_async_engine, AsyncSession
from sqlalchemy.orm import sessionmaker

engine = create_async_engine(settings.DATABASE_URL, echo=settings.DEBUG)
async_session = sessionmaker(engine, class_=AsyncSession, expire_on_commit=False)

async def get_db():
    async with async_session() as session:
        yield session
```

```python
# 路由中使用
@router.get("/")
async def list_items(db: AsyncSession = Depends(get_db)):
    ...
```

## 七、权限校验入口

### 7.1 认证方式

| 方式 | 说明 |
|------|------|
| **JWT Token** | 用户登录后颁发；请求头 `Authorization: Bearer <token>` |
| **微信登录** | 第二版实现；第一版用邮箱/密码 |

### 7.2 权限依赖

```python
# app/core/dependencies.py
from fastapi import Depends, HTTPException, status
from fastapi.security import HTTPBearer, HTTPAuthorizationCredentials

security = HTTPBearer()

async def get_current_user(
    credentials: HTTPAuthorizationCredentials = Depends(security),
    db: AsyncSession = Depends(get_db)
) -> User:
    token = credentials.credentials
    user = await verify_token(token, db)
    if not user:
        raise HTTPException(status_code=401, detail="Invalid token")
    return user

async def require_admin(user: User = Depends(get_current_user)) -> User:
    if not user.is_admin:
        raise HTTPException(status_code=403, detail="Admin required")
    return user
```

## 八、框架最大化利用原则

| 原则 | 说明 |
|------|------|

## 九、统计分析规则与知识融合设计

> 本章定义从「数据分析模板包」（外部知识资产，21 文件 / 7 模块）提取统计规则并融合到后端功能的设计。原则：**提取规则增强功能，不把模板包整体搬入做内容站**（与立项文档「教程站是后期规划」红线一致）。

### 9.1 融合范围

| 模板包模块 | 融合方式 | 落点 | 优先级 |
|---|---|---|---|
| 02 信效度（α/KMO/载荷分档 + 翻车点 TOP10） | 提取规则 | `core/statistics_constants.py` + `diagnoser` 增强 | P0 |
| 05 样本量（效应量档位） | 校准统一 | `generator` + `hypothesis_parser` | P0 |
| 03 差异检验决策树 | 提取规则 | `stats` 方法选择 | P1 |
| 04 回归翻车点 | 提取规则 | `diagnoser` 知识库扩展 | P1 |
| 论文段落模板 | 套用句式 | `reporter` 导出 Word | P1 |
| 01 问卷设计 / 06 可视化 / 07 AI辅助 | **不融合** | 弱相关或定位冲突，留作后期教程站种子 | - |

### 9.2 统计阈值常量化

**现状问题**：[stats.py#L81](../server/app/services/stats.py) 与 [diagnoser.py#L16-L20](../server/app/services/diagnoser.py) 各自硬编码阈值，且 diagnoser 的 prompt 文字又重复写一遍标准，存在三处不同步风险。

**设计**：新增 `app/core/statistics_constants.py`，集中定义分档表与阈值，作为统计判定的唯一来源。

分档标准（来源：模板包「信效度速查表」，基于公开统计学规范）：

| 指标 | 优秀 | 良好 | 可接受 | 不达标 |
|---|---|---|---|---|
| Cronbach's α | ≥0.90 | 0.80–0.89 | 0.70–0.79 | <0.70（0.60–0.69 偏低，删题后再用） |
| KMO | ≥0.90 | 0.80–0.89 | 0.50–0.79 | <0.50 |
| Bartlett p | <0.001 | <0.01 | <0.05 | ≥0.05 |
| 因子载荷 | ≥0.70 | 0.60–0.69 | 0.40–0.59 | <0.40 |
| 累计方差解释率 | ≥70% | 60–69% | 50–59% | <50% |

**KMO 阈值校准决策**：
- 现状：`stats.py` / `diagnoser.py` 用 `kmo >= 0.6` 判定通过
- 问题：比学术通行标准（0.5 可接受）偏严，可能误判 KMO=0.55 为不达标
- 决策：**KMO 通过阈值 0.6 → 0.5**，对齐上表「可接受」下界
- 影响面：`stats.py` passed 判定、`diagnoser.py` THRESHOLDS、诊断 prompt 文字

**接口设计**（实现规格，不含代码）：
```
statistics_constants.py
├── ALPHA_GRADES: [(阈值, 等级, 论文措辞)]  # 如 (0.9,"极佳","信度极好")
├── KMO_GRADES / BARTLETT_GRADES / LOADING_GRADES / VARIANCE_GRADES
├── THRESHOLDS: {"alpha":0.7, "kmo":0.5, "bartlett_p":0.05}  # 通过线
├── grade_alpha(value) -> (等级, 措辞)
└── is_passed(metric, value) -> bool
```
`stats.py` 与 `diagnoser.py` 改为引用本常量，消除硬编码；`diagnoser` prompt 改为注入分档表文本而非重复写死标准。

### 9.3 诊断知识库（翻车点 → issue 匹配）

**现状问题**：[diagnoser.py](../server/app/services/diagnoser.py) 完全依赖 LLM 自由生成 reason/suggestion，无规则兜底，结果不稳定且可能遗漏高频翻车点。

**设计**：将模板包「信效度翻车点 TOP10」结构化为规则库，诊断时先规则匹配、再 LLM 补充。

翻车点 TOP10（来源：模板包「信效度速查表」）：

| # | 翻车点 | 触发条件（可检测） | 建议方向 |
|---|---|---|---|
| 1 | 反向题没反转 | 反向题标记存在但未处理 | 反转计分 |
| 2 | 混维度算 α | — | 按维度分别计算 |
| 3 | 只报整体 α 不报分维度 | — | 补分维度报告 |
| 4 | KMO<0.5 仍强行 EFA | kmo<0.5 | 不宜做因子分析 |
| 5 | 载荷<0.4 不删题 | 载荷<0.4 | 删除该题 |
| 6 | 一题跨多因子 | 跨因子载荷差<0.2 | 按较高载荷归类或删除 |
| 7 | 不报累计方差解释率 | — | 补报累计方差% |
| 8 | 因子数与理论不符 | 提取因子数≠维度数 | 说明或调整 |
| 9 | 样本太少做 EFA | n<100 | 增大样本 |
| 10 | 编造 α/载荷值 | — | 红线，拒绝 |

**结构设计**：新增 `app/services/diagnosis_rules.py`
```
diagnosis_rules.py
├── PITFALLS: [{code, name, detect(metrics)->bool, suggestion}]
└── match_pitfalls(reliability_results, project_meta) -> List[issue]
```
**诊断流程改为**：规则匹配（确定性，必出）→ LLM 补充（生成自然语言 reason）→ 合并去重。规则保证高频问题不漏，LLM 负责语境化表达。

### 9.4 报告分档展示与段落模板

**现状问题**：[reporter.py](../server/app/services/reporter.py) 导出 Word 只输出数值 + passed 布尔，缺分档标签和标准论文措辞，用户拿到的报告需自行组织语言。

**设计**：
1. 报告输出每个指标附加 `(等级, 论文措辞)`，如 α=0.7955 → 「可接受 / 信度可接受」
2. 导出 Word 时套用模板包「论文信效度段落模板（一段式）」生成可直接进论文方法部分的标准段落
3. 报告附录附 1–2 页对应速查表（信效度速查 + 回归翻车点），增强报告可用性（C 方案）

**段落模板**（占位符来自模板包，填入计算结果）：
> 本研究的《`{量表名}`》共 `{维度数}` 个维度 `{题项数}` 个题项。信度检验显示，总量表 Cronbach's α = `{α}`，各维度 α 介于 `{min}`～`{max}`，均大于 0.7，信度可接受。效度检验中，KMO = `{kmo}`，Bartlett 球形检验 p < 0.05，适合做因子分析；主成分分析提取 `{因子数}` 个公因子，累计方差解释率 `{方差%}`，各题载荷 ≥ `{载荷}`，结构效度良好。

### 9.5 效应量档位统一（关键决策）

**三方不一致现状**（必须解决）：

| 来源 | 弱 | 中 | 强 |
|---|---|---|---|
| Cohen 国际标准（模板包）r | 0.10 | 0.30 | 0.50 |
| 立项文档定义 r | 0.20 | 0.40 | 0.60 |
| 代码实现 [generator.py#L15](../server/app/services/generator.py) `STRENGTH_TO_R` | 0.25 | 0.45 | 0.65 |

**决策**：采用「名义值对齐 Cohen + 内部离散化补偿」双层方案。
- **名义值（对外/文档/前端展示）**：弱/中/大 = r 0.1 / 0.3 / 0.5（Cohen 标准，学术规范）
- **生成补偿值（内部）**：r 0.2 / 0.4 / 0.6（= 立项文档值），补偿李克特量表离散化（连续→1-5 整数）造成的相关性衰减
- **修正代码**：`STRENGTH_TO_R` 0.25/0.45/0.65 → 0.2/0.4/0.6（消除代码与立项文档偏差）

**风险评估**：
- 影响面：[generator.py](../server/app/services/generator.py)、[hypothesis_parser.py](../server/app/services/hypothesis_parser.py) prompt 措辞、前端 `strength-selector` 标签
- V4 验收（α≥70%）：补偿值下调后需重新验证达标率，现有 α=0.7955 数据基于 0.25/0.45/0.65 生成，改后预期略降但仍在阈值上方
- 历史数据：已生成 dataset 的 strength 语义不变（仍是 weak/medium/strong 标签），仅数值映射口径变化，不影响存量数据可用性
- 文档同步：立项文档第 32 行需补充「名义值 vs 补偿值」说明

**结构设计**：
```
generator.py
├── STRENGTH_NOMINAL = {"weak":0.1, "medium":0.3, "strong":0.5}  # 对外展示
├── LIKERT_DISCRETIZATION_COMPENSATION = 2.0  # 离散化补偿系数（名义×2 取整）
└── STRENGTH_TO_R = {"weak":0.2, "medium":0.4, "strong":0.6}    # 生成用补偿值
```
前端展示名义档位与 Cohen 解释，生成服务用补偿值，二者分离避免「美化数据」合规嫌疑。

### 9.6 差异检验方法决策树

**现状**：V6 差异检验未做方法自动选择（需用户指定）。

**设计**：提取模板包「差异检验速查表」选检验决策树为规则，`stats` 根据变量类型/组数自动推荐方法：

| 自变量 | 因变量 | 组数 | 推荐方法 |
|---|---|---|---|
| 分类 | 连续 | 2 | 独立样本 t |
| 分类 | 连续 | ≥3 | 单因素 ANOVA |
| 分类 | 分类 | - | 卡方 |
| 连续 | 连续 | - | Pearson 相关 |
| 连续×多 | 连续 | - | 线性回归 |

优先级 P1，作为 V6 增强项，不阻塞当前 MVP。

### 9.7 数据库影响

本章设计**不涉及表结构变更**：
- 阈值常量、翻车点规则、段落模板均为代码常量，不入库
- 报告分档标签为计算结果，随报告一起存入现有 `reports` / `reliability_results` 表（可选新增 `grade` 字段，但非必须——可由 `alpha` 值实时计算分档）
- 若需持久化分档标签便于查询/统计，可后续在 `reliability_results` 加 `grade VARCHAR(8)` 字段（须走数据库变更流程：设计文件 → 评审 → 迁移）

## 十二、SQLite 开发环境注意事项

> 生产环境用 PostgreSQL，以下问题仅影响开发环境。

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| **并发读写锁定** | SQLite 默认不允许并发写 | `database.py` 配置 WAL 模式（`PRAGMA journal_mode=WAL`）+ busy_timeout |
| **WAL 读快照** | 一个连接的读快照在事务期间固定，外部进程写入后需重新连接才能看到新数据 | 验证脚本查 DB 前需 `conn.close()` 重连 |
| **UUID 存储格式** | `sqlalchemy.Uuid` 在 SQLite 上用 `.hex`（32 位无连字符）存储 | 外部脚本直连 DB 插入 UUID 时必须用 `.hex` 格式 |
| **历史脏数据** | 从 `postgresql.UUID` 迁移到 `sqlalchemy.Uuid` 后，早期用 `str(uuid)` 插入的行无法被 `db.get()` 命中 | 将历史 id UPDATE 为 `.hex` 格式 |
| **alembic.ini 编码** | Windows configparser 默认 GBK 解码失败 | alembic.ini 不能包含中文注释；env.py 需 `sys.path.insert` 加入 server 目录 |
| **系统代理干扰** | Windows 注册表代理被 httpx/openai SDK 自动读取 | 启动 uvicorn 前 `$env:NO_PROXY="api.deepseek.com"` |
